Build your own vanilla Javascript SPA router

28.07.2020 — Javascript, router, SPA

SPA(Single Page Application) router

서버로부터 완전히 새로운 페이지를 불러오지 않고 현재 페이지를 web server로부터 받은 data를 이용해 페이지를 동적으로 변경하는 Web Application을 SPA라고 한다.

SPA 이전의 전통적인 web site들은 사용자가 요청하는 uri에 해당하는 페이지(html 파일)를 web server로부터 전달받아 보여준다. URI에 해당하는 페이지(html 파일)를 보여줬기 때문에 router의 역할이 중요하지 않았다.

한편, SPA는 필요한 HTML, CSS, Javascript static 파일들을 서버로부터 처음의 다운로드 받고, 이후에 필요한 resource들은 ajax를 이용해 가져온다.

SPA에서 사용자는 아래의 몇 가지 방법으로 새로운 페이지를 요청할 수 있다.

  • 사용자가 browser 주소창에 새로운 URL을 입력하는 경우
  • 사용자가 browser의 뒤로가기, 앞으로가기 버튼을 누르는 경우
  • 사용자가 페이지의 link를 눌러 페이지를 이동하는 경우

한편, SPA는 전통적인 web site와 달리 URL이 바뀐다고 해서 web server로부터 HTML, CSS 등 모든 static 파일들을 새롭게 다운받지 않는다. 여기서 router의 활약이 필요하게 된다. 사용자가 요청한 URI에 맞는 자원(페이지)을 렌더링 할 수 있게 연결하는 것이 SPA router의 임무다.

HTML 5 history API를 활용해 간단하게 vanilla Javascript SPA router를 구현해보았다.

Vanilla Javascript SPA router

SPA structure

먼저 index.html, index.js 로 SPA의 structure를 잡았다.

1<body>
2  <div id="root"></div>
3  <script src="src/index.js"></script>
4</body>
1import App from "./App";
2

3const root = document.querySelector("#root");
4root.innerHTML = App();

Simple Application

간단한 예제로 navigation bar의 link를 클릭하면, 주소가 변경되고 그 주소에 맞는 view가 보여지는 간단한 기능을 하는 Application이다.

1export default function App() {
2  function render() {
3    const html = `
4      <ul class="nav-bar">
5        <li class="nav-bar-item">
6          <a class="link" id="/">home</a>
7        </li>
8        <li class="nav-bar-item">
9          <a class="link" id="about">about</a>
10        </li>
11        <li class="nav-bar-item">
12          <a class="link" id="posts">posts</a>
13        </li>                
14      </ul>
15      <h1>Vanilla Javascript Router</h1>
16    `;
17    const $app = document.querySelector("#app");
18    $app.innerHTML = html;
19

20    const $nav = document.querySelector("ul.nav-bar");
21    $nav.addEventListener("click", onLinkClick);
22  }
23

24  setTimeout(render, 0);
25

26  return `<div id="app"></div>`;
27}

이제 해야할 일은 아래와 같다.

  • router를 구현해, 주소에 알맞는 view를 nav에 더해 페이지를 렌더링하기
  • onLinkClick() 함수를 구현해, 새로운 주소로 이동시키기

Router

현재 App은 한 가지 view만 사용자에게 보여준다. 이를 router를 이용해 URL과 view를 mapping해주자.

1class Router {
2  constructor() {
3    this.routes = {
4      "": this.HomePage(),
5      about: this.AboutPage(),
6      posts: this.PostsPage()
7    }
8  }
9}
10

11export const router = new Router()

constructor() 에서 URL과 view를 key, value형태로 가지고 있는 routes object를 생성한다. HomePage(), AboutPage(), PostsPage()는 각 주소에 알맞는 view를 리턴하는 함수다.

마지막으로, 현재 주소를 인자로 받아 그에 알맞는 view를 리턴하는 method들을 구현한다.

1class Router {
2  // ...
3  getCurrentPath() {
4    return window.location.pathname.replace("/", "");
5  }
6

7  mapURItoView() {
8    return this.routes[this.getCurrentPath()];
9  }
10  // ...
11}

Let the router work

이제 구현한 router를 App에 적용시키자.

1import { router } from './router'
2export default function App() {
3  //...
4  function render() {
5    const nav = `
6      <ul class="nav-bar">
7        ...           
8      </ul>
9    `;
10    const html = nav.concat(router.mapURItoView());
11    //...
12  }
13  //...
14}

router 덕분에 주소에 맞는 view를 렌더링 할 수 있게 되었다. 이제 onLinkClick() event handler를 구현해서 링크를 클릭했을 때 주소를 변경시켜주자.

History API

HTML5의 history API는 pushState(), popState() 등의 method로 브라우저의 history를 관리할 수 있도록 도와준다. push, pop이라는 단어에서 알 수 있듯이 stack을 연상시킬 수 있다. 새로운 주소로 이동하면서 history란 stack에 push하고, 뒤로가기를 하면서 history stack에서 pop하는 것으로 자료구조와 연관지어 이해했다.

이 history API를 이용해 사용자가 link를 click했을 때, 주소를 이동시켜주는 기능을 구현했다.

1function onLinkClick(e) {
2  if (e.target.className === "link") {
3    e.preventDefault();  // prevent it since this handler is attatched to container. Event Delegation.
4    const path = e.target.id;
5    window.history.pushState(null, null, path); // move to new path
6    render();
7  }
8}

a tag의 id에 새로 이동할 path의 정보를 넣어두었기 때문에 위와 같이 pushState()를 이용해 새로운 주소로 이동할 수 있다. pushState() 함수의 앞의 두 parameter는 각각 state와 title인데 우리 구현에서는 필요없기 때문에 null을 넣어주었다.

1window.addEventListener("popstate", () => {
2  render();
3});

popstate event가 발생하면 App을 다시 render하게 함으로써 사용자가 뒤로가기, 앞으로가기 버튼으로 주소를 변경하여도 대응할 수 있게 된다.

Conclusion

Vanilla Javascript router를 구현하면서 몇 가지 소득이 있었다.

우선은 Single Page Application의 동작 방식에 대해 다시 한 번 생각해보는 기회가 되었다. 웹 개발을 시작한지 얼마되지 않았기 때문에 전통적인 web site와 동작방식을 직접 접해본 기억이 없다. 둘을 비교하면서 SPA가 등장한 배경까지도 이해할 수 있었다.

두 번째로는 앞으로 사용하게 될 Javascript Framework의 router의 본질을 알게 되었다. SPA에서 Router의 역할이 무엇인지 알았기 때문에, React, Vue 등의 framework의 router를 접해도 당황하지 않고 금방 사용할 수 있을 것 같다.
