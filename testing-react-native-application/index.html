Testing React Native Application

29.01.2021 — React Native, Test

React Native Application에 테스트 자동화를 적용하게 된 이야기를 정리합니다

여러분의 React/React-Native 프로젝트에 테스트 자동화가 구축되어 있나요? Test Driven Development(이하 TDD)를 하는 팀도 있을 것이고, TDD는 아니지만 테스트 코드를 구축하고 있는 팀도 있을 것이며 테스팅을 전혀 하지 않는 팀도 있을 것입니다. Frontend 또는 Client 프로젝트의 UI 컴포넌트 테스트, Rendering 테스트의 경우 그 효용성, 효율성에 있어 갑론을박도 있습니다.

그럼에도 불구하고 어떤 이유로 테스팅을 도입하게 되었고 어떤 것을, 어떤 방법으로 테스트하게 되었는지 이 글을 통해 정리해보고자 합니다.

Why test

React Native reference 에서는 테스트 코드의 효과를 아래와 같이 나열하고 있습니다.

  • 기능 추가 후에 Application의 동작을 검증할 수 있다.
  • 기존 코드 리팩토링 후에 Application의 동작을 검증할 수 있다.
  • dependency library를 업데이트 후에 Application의 동작을 검증할 수 있다.
  • bug case를 테스트코드로 작성한 뒤 그것을 pass 하도록 코드를 수정해 bug fix를 할 수 있다.
  • 미래에 join하는 팀원들에게 documentation을 제공해줄 수 있다.
  • QA cost를 절감할 수 있다.

제 경우에는 기존 컴포넌트를 리팩토링 하면서 테스트 코드의 필요성을 크게 느꼈습니다. 프로젝트 곳곳에서 활용되고 있는 컴포넌트의 스타일을 계산하는 로직에 대한 리팩토링이었기 때문에 영향범위가 매우 컸습니다. 테스트 코드 없이 리팩토링 전후의 렌더링되는 스타일을 직접 눈으로 확인하는 것은 불가능에 가까웠기 때문에 테스트라는 장치가 필요했습니다.

지금 당장은 안전 장치로서 가치가 있지만 테스트 코드가 미래에 줄 가치에 대해서도 고려했습니다. 미래에 프로젝트에 합류할 동료를 위한 문서화의 역할을 할 수 있을 것이라는 기대와 개발자로서 보다 testable한 code를 작성하도록 노력하게 만들 것이라는 기대로 시작하게 되었습니다.

What to test

이런 이유에서 테스트 코드를 작성하게 되었기 때문에 자연스럽게 UI 컴포넌트에 대한 테스트 코드를 가장 먼저 추가하게 되었습니다. 컴포넌트 테스트에서는 유저 interaction에 대한 컴포넌트의 변경을 중점적으로 테스트하였습니다. 반면, 비즈니스 로직에 의해 쉽게 변경될 수 있는 컴포넌트의 state나, 부모로부터 전달받는 props에 대해서는 검증하지 않았습니다.

UI 컴포넌트 뿐만 아니라, utility 함수들에 대해 단위테스트를 추가했으며 redux reducer와 action creator(async action creator)에 대해서도 테스트 코드를 작성했습니다.

How to test

테스트 프레임워크는 Jest를, 라이브러리는 react-native-testing-library를 활용했습니다. 또한 testing-library/jest-native의 matcher를 이용해서 assertion을 했습니다.

UI component test

<Text /> 컴포넌트를 <TouchableOpacity /> 로 감싼 간단한 <ButtonPrimary /> 컴포넌트를 만들었습니다. 이 컴포넌트는 inverted 여부에 따라 2가지 스타일로 렌더되며 press했을 때, props로 전달한 callback을 실행하는 기능을 합니다.

1// ButtonPrimary.tsx - before refactoring
2import { PALETTE } from 'constants/styles';
3import React from 'react';
4import { If, Else, Then } from 'react-if';
5import { Text, TouchableOpacity } from 'react-native';
6import tailwind from 'tailwind-rn';
7

8interface Props {
9  label: string;
10  onPress: () => void;
11  inverted?: boolean;
12    disabled?: boolean;
13}
14

15export const ButtonPrimary: React.FC<Props> = ({
16  label,
17  onPress,
18  inverted = false,
19    disabled = false,
20}) => {
21  return (
22    <If condition={!inverted}>
23      <Then>
24        <TouchableOpacity
25          style={{
26            ...tailwind('w-full py-2 rounded-lg'),
27            backgroundColor: PALETTE.BEIGE,
28          }}
29          onPress={onPress}
30                    disabled={disabled}
31          activeOpacity={0.7}>
32          <Text
33            style={{
34              ...tailwind('text-lg text-center text-white'),
35              fontFamily: 'SpoqaHanSansNeo-Bold',
36            }}
37            allowFontScaling={false}
38            selectable={false}>
39            {label}
40          </Text>
41        </TouchableOpacity>
42      </Then>
43      <Else>
44        <TouchableOpacity
45          style={tailwind('w-full py-2 rounded-lg bg-white')}
46          onPress={onPress}
47                    disabled={disabled}
48          activeOpacity={0.7}>
49          <Text
50            style={{
51              ...tailwind('text-lg text-center'),
52              color: PALETTE.BEIGE,
53              fontFamily: 'SpoqaHanSansNeo-Bold',
54            }}
55            allowFontScaling={false}
56            selectable={false}>
57            {label}
58          </Text>
59        </TouchableOpacity>
60      </Else>
61    </If>
62  );
63};

리팩토링 이전에 사용자의 press interaction에 따라 버튼 컴포넌트가 어떻게 변경되는지 specification을 정의 했으며, inverted-not inverted 두 가지 스타일에 대한 test case도 만들었습니다.

1// ButtonPrimary.spec.js
2import React from 'react';
3import { render, fireEvent } from '@testing-library/react-native';
4import { ButtonPrimary } from './ButtonPrimary';
5import { PALETTE } from 'constants/styles';
6

7describe('ButtonPrimary', () => {
8  const mockCb = jest.fn();
9

10  afterEach(() => {
11    mockCb.mockClear();
12  });
13

14  it('when pressed, executes callback function', () => {
15    const { getByTestId } = render(
16      <ButtonPrimary label="submit" onPress={mockCb} />,
17    );
18    const button = getByTestId('button-primary');
19

20    fireEvent.press(button);
21

22    expect(mockCb).toHaveBeenCalledTimes(1);
23  });
24

25  it('when disabled, it is not able to execute callback function', () => {
26    const { getByTestId } = render(
27      <ButtonPrimary label="submit" onPress={mockCb} disabled={true} />,
28    );
29    const button = getByTestId('button-primary');
30

31    fireEvent.press(button);
32

33    expect(mockCb).not.toHaveBeenCalled();
34  });
35

36  it('when inverted, the label is beige and background is white', () => {
37    const { getByTestId, getByText } = render(
38      <ButtonPrimary label="submit" onPress={mockCb} inverted={true} />,
39    );
40    const button = getByTestId('button-primary');
41

42    expect(getByText('submit')).toHaveStyle({ color: PALETTE.BEIGE });
43    expect(button).toHaveStyle({ backgroundColor: PALETTE.WHITE });
44  });
45

46  it('when not inverted, the label is white and background is beige', () => {
47    const { getByTestId, getByText } = render(
48      <ButtonPrimary label="submit" onPress={mockCb} />,
49    );
50    const button = getByTestId('button-primary');
51

52    expect(getByText('submit')).toHaveStyle({ color: PALETTE.WHITE });
53    expect(button).toHaveStyle({ backgroundColor: PALETTE.BEIGE });
54  });
55});

테스트 코드라는 안전장치가 생기고 나니 안심하고 리팩토링을 진행할 수 있었습니다.

1// ButtonPrimary.tsx
2import { PALETTE } from 'constants/styles';
3import React from 'react';
4import {
5  StyleProp,
6  StyleSheet,
7  Text,
8  TouchableOpacity,
9  ViewStyle,
10} from 'react-native';
11import tailwind from 'tailwind-rn';
12

13interface Props {
14  label: string;
15  onPress: () => void;
16  disabled?: boolean;
17  inverted?: boolean;
18  testID?: string;
19}
20

21export const ButtonPrimary: React.FC<Props> = ({
22  label,
23  onPress,
24  disabled = false,
25  inverted = false,
26  testID = 'button-primary',
27}) => {
28  return (
29    <TouchableOpacity
30      style={StyleSheet.compose(
31        tailwind('w-full py-2 rounded-lg') as StyleProp<ViewStyle>,
32        inverted
33          ? { backgroundColor: PALETTE.WHITE }
34          : { backgroundColor: PALETTE.BEIGE },
35      )}
36      onPress={onPress}
37      disabled={disabled}
38      activeOpacity={0.7}
39      testID={testID}>
40      <Text
41        style={StyleSheet.compose(
42          {
43            ...tailwind('text-lg text-center'),
44            fontFamily: 'SpoqaHanSansNeo-Bold',
45          },
46          inverted ? { color: PALETTE.BEIGE } : { color: PALETTE.WHITE },
47        )}
48        allowFontScaling={false}
49        selectable={false}>
50        {label}
51      </Text>
52    </TouchableOpacity>
53  );
54};

Unit test of utility functions

Utility 함수의 경우 testable 하게 작성했다면 어렵지 않게 테스트 코드를 작성할 수 있습니다.

1// util.ts
2export const removeHTMLBoldTag = (html: string) =>
3  html.replace(/<b>/g, '').replace(/<\/b>/g, '');
4

5// util.spec.js
6const { removeHTMLBoldTag } = require("utils");
7

8describe('removeHTMLBoldTag', () => {
9  it('cleans any HTML <b> tag in markup', () => {
10    const html = `<p>Hello <b>World</b></p>`;
11    expect(removeHTMLBoldTag(html)).toEqual(`<p>Hello World</p>`)
12  })
13

14  it('cleans unpaired HTML <b> tags', () => {
15    const html = `<p><b>Hello <b>World</b></p>`;
16    expect(removeHTMLBoldTag(html)).toEqual(`<p>Hello World</p>`)
17  })
18

19  it('ignores line break', () => {
20    const html = `<p>Hello <br><b>World</b></p>`;
21    expect(removeHTMLBoldTag(html)).toEqual(`<p>Hello <br>World</p>`)
22  })
23})

Reducer and action creators test

redux reducer와 action creator에 대한 샘플 테스트 코드는 여기에서 확인할 수 있습니다.

When to test

테스트에 대해 필요성을 느끼고, 어떤 테스트를 어떻게 할지까지 정했다면 남은건 언제 테스트 코드를 작성할지 정하는 일입니다. TDD가 익숙하다면 writing a failing test - make the test pass - refactor의 순서로 테스트 코드를 작성할 수 있을 것입니다. 그렇지 않은 경우 언제 테스트를 작성하는게 좋을까요?

테스팅을 하지 않던 프로젝트에 테스팅을 적용한다는 상황을 가정할 때, 개인적으로는 기존 코드의 경우 리팩토링 이전, 새로운 코드의 경우 기능 구현 이후가 적절하다고 생각합니다. 리팩토링 이전에 테스트 코드를 작성하면 개발자에게 안전장치를 제공할 수 있으며 새로운 기능을 개발한 뒤 테스트 코드를 작성하면 미래에 유지보수할 때 문서화 역할을 하며 생산성을 높여줄 수 있다고 생각하기 때문입니다.

그렇다면 작성한 테스트코드로 언제 테스트를 하는 것이 좋을까요? Application의 안정성을 위해서는 자주하면 좋다고 생각합니다. 개인적으로 commit전에 pre-commit hook을 이용해 test를 실행하고, Application build script에 테스트 실행 script를 추가하여 검증하는 방법을 사용하고 있습니다.

Conclusion

테스트가 안정적인 Application을 만들어가는데 도움을 준다는 것은 분명한 사실입니다. 다만 테스트를 적용할지 고민하는 것은 테스트 코드를 작성하는데 들어가는 비용 때문일 것입니다. 숙련되지 않은 상태에서 TDD를 고집하거나, 테스트 커버리지 수치에 목 매다는것보다는 Benefit-Cost를 고려하면서 경제적으로 테스트코드를 작성해나가는 것이 필요하지 않을까 생각하며 글을 줄입니다.
